# How Portainer Kompose Deployments Worked (Removed in 2.17)

Author: [nawazdhandala](https://www.github.com/nawazdhandala)

Tags: Portainer, Kompose, Docker Compose, Kubernetes, Deprecated, History

Description: A historical overview of Portainer's Kompose-based Docker Compose to Kubernetes conversion feature, removed in Portainer 2.17, and what replaced it.

---

Portainer 2.16 and earlier included an integration with Kompose - a tool that converts Docker Compose files into Kubernetes manifests. This allowed operators to paste a Docker Compose definition and have Portainer deploy it to a Kubernetes cluster after automatic conversion. The feature was removed in 2.17 in favor of native Kubernetes manifest and Helm chart support.

## What Was Kompose?

Kompose is an open-source CNCF project that translates Docker Compose service definitions into Kubernetes Deployments, Services, and other objects. It handles the conceptual mapping between Compose and Kubernetes constructs.

## How Portainer Used Kompose

When deploying a stack to a Kubernetes environment in Portainer 2.16 and earlier, operators could:

1. Select the Kubernetes environment
2. Enable the "Kompose" toggle in the stack deployment UI
3. Paste their existing Docker Compose YAML

Portainer would run Kompose internally and deploy the resulting Kubernetes manifests.

A Docker Compose file like this:

```yaml
# docker-compose.yml

version: "3"
services:
  webapp:
    image: nginx:1.25
    ports:
      - "80:80"
    environment:
      - APP_ENV=production
  database:
    image: postgres:16
    environment:
      - POSTGRES_PASSWORD=secret
    volumes:
      - db-data:/var/lib/postgresql/data
volumes:
  db-data:
```

Would be converted by Kompose to Kubernetes objects:

```yaml
# Generated by Kompose - Deployment for webapp
apiVersion: apps/v1
kind: Deployment
metadata:
  name: webapp
spec:
  replicas: 1
  selector:
    matchLabels:
      app: webapp
  template:
    spec:
      containers:
        - name: webapp
          image: nginx:1.25
          ports:
            - containerPort: 80
          env:
            - name: APP_ENV
              value: production
---
# Service for webapp
apiVersion: v1
kind: Service
metadata:
  name: webapp
spec:
  selector:
    app: webapp
  ports:
    - port: 80
      targetPort: 80
```

## Why Kompose Conversions Were Unreliable

The Kompose conversion had significant limitations that made it impractical for production use:

1. **Named volumes** - Kubernetes has no direct equivalent; PersistentVolumeClaims have different semantics
2. **Networks** - Docker networks don't map directly to Kubernetes networking
3. **Healthchecks** - Compose healthchecks aren't directly converted to Kubernetes liveness/readiness probes
4. **Dependencies** - `depends_on` has no Kubernetes equivalent; init containers or application-level retry logic is needed
5. **Secrets/configs** - Compose secrets map loosely to Kubernetes Secrets

## Why It Was Removed

- **Conversion quality** - The generated manifests often required manual editing before they were production-ready
- **Wrong abstraction** - Kubernetes resources need to be designed for Kubernetes, not converted from Compose
- **Better workflow** - The recommended path is to write native Kubernetes manifests or use Helm charts from the start

## What Replaced It

Portainer 2.17+ replaced Kompose integration with first-class Kubernetes manifest and Helm support:

- **Manifest deployments** - Deploy raw YAML/JSON Kubernetes manifests directly in Portainer
- **Helm chart browser** - Browse, install, and manage Helm charts from registered chart repositories
- **Helm values editor** - Customize Helm values in the Portainer UI
- **GitOps support** - Deploy manifests from Git repositories

For teams migrating Compose workloads to Kubernetes, the recommended approach is:

```bash
# Use Kompose externally as a starting point, then refine
kompose convert -f docker-compose.yml -o ./k8s-manifests/

# Review and manually adjust the generated manifests
# Add proper resource limits, health checks, and security contexts
# Then deploy via Portainer's manifest deployment feature
```

## Summary

The Kompose integration was a well-intentioned feature that tried to lower the barrier to Kubernetes adoption for Docker Compose users. Its removal acknowledged that automated conversion produces second-rate Kubernetes manifests, and that better tooling (Helm, native manifests, GitOps) serves operators better in practice.
