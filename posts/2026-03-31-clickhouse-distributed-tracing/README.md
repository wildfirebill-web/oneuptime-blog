# How to Use Distributed Tracing with ClickHouse

Author: [nawazdhandala](https://www.github.com/nawazdhandala)

Tags: ClickHouse, Distributed Tracing, OpenTelemetry, Observability, Performance, Query

Description: Learn how to correlate ClickHouse query execution across nodes using query_id propagation, opentelemetry_span_log, and how to store and query distributed trace data in ClickHouse itself.

---

Distributed tracing in the context of ClickHouse has two distinct meanings: tracing ClickHouse's own query execution across a distributed cluster using the built-in OpenTelemetry span integration, and using ClickHouse as a storage backend for traces generated by your application services. This guide covers both.

## Part 1: Tracing ClickHouse Query Execution

### How ClickHouse Propagates query_id

Every query in ClickHouse has a `query_id`. When a `Distributed` table fans out sub-queries to remote shards, it propagates the `initial_query_id` so you can correlate the top-level query with all its remote executions:

```sql
-- Set a custom query_id for easy correlation
SET query_id = 'my-trace-abc123';

SELECT
    count(),
    uniq(user_id)
FROM distributed_events
WHERE event_date = today();

-- After the query completes, find all sub-queries on all shards
SELECT
    host_name,
    query_id,
    initial_query_id,
    is_initial_query,
    query_duration_ms,
    read_rows,
    formatReadableSize(read_bytes) AS bytes_read,
    substr(query, 1, 100)          AS query_preview
FROM clusterAllReplicas('your_cluster', system.query_log)
WHERE
    initial_query_id = 'my-trace-abc123'
    AND type = 'QueryFinish'
ORDER BY query_duration_ms DESC;
```

### Enabling OpenTelemetry Tracing

ClickHouse can emit OpenTelemetry-compatible spans for every query. Enable the `opentelemetry_span_log` table:

```bash
sudo tee /etc/clickhouse-server/config.d/opentelemetry.xml > /dev/null <<'EOF'
<clickhouse>
    <opentelemetry_span_log>
        <database>system</database>
        <table>opentelemetry_span_log</table>
        <flush_interval_milliseconds>7500</flush_interval_milliseconds>
        <engine>
            ENGINE = MergeTree
            PARTITION BY toYYYYMM(finish_date)
            ORDER BY (finish_date, finish_time_us, trace_id)
            TTL finish_date + INTERVAL 30 DAY
        </engine>
    </opentelemetry_span_log>
</clickhouse>
EOF

sudo systemctl restart clickhouse-server
```

### Triggering a Traced Query

Pass OpenTelemetry headers to make ClickHouse participate in an existing trace:

```bash
# Use curl to send a query with OpenTelemetry trace context
TRACE_ID="4bf92f3577b34da6a3ce929d0e0e4736"
SPAN_ID="00f067aa0ba902b7"
PARENT_SPAN_ID="00f067aa0ba902b7"

curl -s "http://localhost:8123/" \
  -H "X-OpenTelemetry-Traceparent: 00-${TRACE_ID}-${SPAN_ID}-01" \
  --data-urlencode "query=SELECT count() FROM system.tables"
```

Or set tracing parameters at the query level:

```sql
SET opentelemetry_start_new_trace = 1;
SET opentelemetry_trace_id = '4bf92f3577b34da6a3ce929d0e0e4736';
SET opentelemetry_span_id  = '00f067aa0ba902b7';

SELECT count(), uniq(user_id) FROM events WHERE event_date = today();
```

### Querying opentelemetry_span_log

```sql
-- Find all spans for a specific trace
SELECT
    trace_id,
    span_id,
    parent_span_id,
    operation_name,
    start_time_us,
    finish_time_us,
    (finish_time_us - start_time_us) / 1000 AS duration_ms,
    attribute
FROM system.opentelemetry_span_log
WHERE trace_id = '4bf92f3577b34da6a3ce929d0e0e4736'
ORDER BY start_time_us;

-- Find slow spans across all traces
SELECT
    operation_name,
    count()                                              AS span_count,
    avg((finish_time_us - start_time_us) / 1000)       AS avg_ms,
    max((finish_time_us - start_time_us) / 1000)       AS max_ms,
    quantile(0.99)((finish_time_us - start_time_us) / 1000) AS p99_ms
FROM system.opentelemetry_span_log
WHERE finish_date >= today() - 1
GROUP BY operation_name
ORDER BY avg_ms DESC;
```

## Part 2: Storing Application Traces in ClickHouse

ClickHouse is an excellent backend for storing OpenTelemetry trace data due to its columnar compression, fast aggregations, and time-series partitioning.

### Creating the Traces Table

```sql
CREATE TABLE otel_traces
(
    trace_id          FixedString(16),
    span_id           FixedString(8),
    parent_span_id    FixedString(8),
    trace_state       String,
    span_name         LowCardinality(String),
    span_kind         LowCardinality(String),
    service_name      LowCardinality(String),
    resource_attributes Map(LowCardinality(String), String),
    span_attributes   Map(LowCardinality(String), String),
    duration_nano     UInt64,
    status_code       LowCardinality(String),
    status_message    String,
    events_name       Array(String),
    events_timestamp  Array(DateTime64(9)),
    events_attributes Array(Map(LowCardinality(String), String)),
    links_trace_id    Array(FixedString(16)),
    links_span_id     Array(FixedString(8)),
    links_attributes  Array(Map(LowCardinality(String), String)),
    timestamp         DateTime64(9),
    end_timestamp     DateTime64(9)
)
ENGINE = MergeTree
PARTITION BY toDate(timestamp)
ORDER BY (service_name, span_name, toUnixTimestamp(timestamp))
TTL toDate(timestamp) + INTERVAL 30 DAY
SETTINGS index_granularity = 8192;
```

### Shipping Traces via OpenTelemetry Collector

Configure the OpenTelemetry Collector to export traces to ClickHouse:

```yaml
# otel-collector-config.yaml
receivers:
  otlp:
    protocols:
      grpc:
        endpoint: 0.0.0.0:4317
      http:
        endpoint: 0.0.0.0:4318

processors:
  batch:
    timeout: 1s
    send_batch_size: 1024

exporters:
  clickhouse:
    endpoint: tcp://clickhouse:9000
    database: default
    traces_table_name: otel_traces
    timeout: 5s
    retry_on_failure:
      enabled: true
      initial_interval: 5s
      max_interval: 30s
      max_elapsed_time: 300s

service:
  pipelines:
    traces:
      receivers: [otlp]
      processors: [batch]
      exporters: [clickhouse]
```

### Querying Application Traces

Find the slowest spans for a specific service:

```sql
SELECT
    span_name,
    service_name,
    count()                          AS span_count,
    avg(duration_nano / 1e6)         AS avg_ms,
    max(duration_nano / 1e6)         AS max_ms,
    quantile(0.95)(duration_nano / 1e6) AS p95_ms
FROM otel_traces
WHERE
    toDate(timestamp) >= today() - 7
    AND service_name = 'api-gateway'
GROUP BY span_name, service_name
ORDER BY avg_ms DESC
LIMIT 20;
```

Reconstruct a full trace:

```sql
SELECT
    hex(trace_id)     AS trace_id_hex,
    hex(span_id)      AS span_id_hex,
    hex(parent_span_id) AS parent_hex,
    service_name,
    span_name,
    timestamp,
    round(duration_nano / 1e6, 2) AS duration_ms,
    status_code,
    span_attributes
FROM otel_traces
WHERE trace_id = unhex('4bf92f3577b34da6a3ce929d0e0e4736')
ORDER BY timestamp;
```

Find traces with errors:

```sql
SELECT
    hex(trace_id) AS trace_id,
    service_name,
    span_name,
    status_message,
    span_attributes['error'] AS error_detail,
    timestamp
FROM otel_traces
WHERE
    toDate(timestamp) >= today() - 1
    AND status_code = 'STATUS_CODE_ERROR'
ORDER BY timestamp DESC
LIMIT 50;
```

## Summary

ClickHouse supports distributed tracing at two levels. For its own internal query execution, it propagates `initial_query_id` across shards so you can correlate sub-queries across a cluster, and its `opentelemetry_span_log` table stores OpenTelemetry spans for queries when tracing is enabled. For application traces, ClickHouse serves as an efficient storage backend through the OpenTelemetry Collector ClickHouse exporter, enabling high-volume trace ingestion with fast aggregation queries over service name, span name, and duration percentiles.
