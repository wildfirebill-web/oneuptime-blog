# How to Use .auto.tfvars for Automatic Variable Loading in OpenTofu

Author: [nawazdhandala](https://www.github.com/nawazdhandala)

Tags: OpenTofu, Variables, Auto.tfvars, Configuration Files, Infrastructure as Code, DevOps

Description: Learn how to use .auto.tfvars files for automatic variable loading in OpenTofu without needing to specify -var-file flags.

---

Files ending in `.auto.tfvars` or `.auto.tfvars.json` are automatically loaded by OpenTofu in addition to `terraform.tfvars`. This naming pattern is useful for organizing variables into logical groups that are always applied, or for injecting variables from external tooling without modifying the base `terraform.tfvars`.

---

## How .auto.tfvars Works

OpenTofu automatically loads these files in alphabetical order:
1. `terraform.tfvars`
2. `terraform.tfvars.json`
3. All `*.auto.tfvars` files (alphabetically)
4. All `*.auto.tfvars.json` files (alphabetically)

```text
project/
├── terraform.tfvars          # base values (loaded first)
├── networking.auto.tfvars    # networking-specific vars (auto-loaded)
├── security.auto.tfvars      # security-specific vars (auto-loaded)
└── tags.auto.tfvars          # tag values (auto-loaded)
```

## Example: Splitting Variables into Logical Groups

```hcl
# networking.auto.tfvars - network configuration

vpc_cidr_block    = "10.0.0.0/16"
public_subnets    = ["10.0.1.0/24", "10.0.2.0/24"]
private_subnets   = ["10.0.3.0/24", "10.0.4.0/24"]
availability_zones = ["us-east-1a", "us-east-1b"]
```

```hcl
# tags.auto.tfvars - common tags applied to all resources
common_tags = {
  ManagedBy   = "opentofu"
  Repository  = "github.com/myorg/infrastructure"
  CostCenter  = "platform-engineering"
}
```

```hcl
# security.auto.tfvars - security configuration
allowed_ssh_cidrs = ["10.0.0.0/8"]
enable_waf        = true
ssl_policy        = "ELBSecurityPolicy-TLS13-1-2-2021-06"
```

## CI/CD Use Case: Injecting Variables Without File Modification

CI/CD pipelines can generate an `.auto.tfvars` file during the pipeline run to inject build-specific variables:

```bash
#!/bin/bash
# generate-ci-vars.sh - generates .auto.tfvars from CI environment

cat > ci-generated.auto.tfvars << EOF
# Auto-generated by CI pipeline - do not edit
image_tag      = "${CI_COMMIT_SHA:-latest}"
deploy_time    = "$(date -u +%Y-%m-%dT%H:%M:%SZ)"
pipeline_id    = "${CI_PIPELINE_ID:-local}"
EOF

echo "Generated ci-generated.auto.tfvars"
cat ci-generated.auto.tfvars
```

```yaml
# GitHub Actions workflow
- name: Generate CI vars
  run: |
    cat > ci-generated.auto.tfvars << EOF
    image_tag   = "${{ github.sha }}"
    deploy_time = "$(date -u +%Y-%m-%dT%H:%M:%SZ)"
    EOF

- name: Plan
  run: tofu plan  # ci-generated.auto.tfvars is auto-loaded
```

## Alphabetical Loading Order Matters

When multiple `.auto.tfvars` files set the same variable, the later file (alphabetically) wins:

```hcl
# a-defaults.auto.tfvars
region = "us-east-1"  # loaded first

# z-overrides.auto.tfvars
region = "us-west-2"  # loaded last - this value wins
```

```bash
# Verify which value is active
tofu console
> var.region
"us-west-2"
```

## Summary

`.auto.tfvars` files are automatically loaded by OpenTofu, making them useful for splitting configuration into logical groups or injecting CI/CD-generated values without modifying existing files. Files are loaded alphabetically, so later files override earlier ones for the same variable. Use this pattern for team-specific defaults, per-environment tag files, or CI pipeline variable injection.
