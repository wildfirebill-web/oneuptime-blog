# How to Create a MySQL Database Snapshot for Testing

Author: [nawazdhandala](https://www.github.com/nawazdhandala)

Tags: MySQL, Snapshot, Testing, Backup, mysqldump

Description: Learn how to create and restore MySQL database snapshots for testing using mysqldump, LVM snapshots, and Docker-based baseline images for fast test environment resets.

---

## What Is a Test Snapshot?

A test database snapshot is a point-in-time copy of a database in a specific state - typically after migrations are applied and with representative fixture data loaded. Restoring from a snapshot is faster than re-running all migrations and seeders, especially for databases with complex schemas.

## Method 1 - mysqldump Snapshot

mysqldump creates a portable SQL dump that can be restored to any MySQL instance:

```bash
# Create snapshot after migrations and seeding
mysqldump \
  -h 127.0.0.1 \
  -u testuser \
  -ptestpass \
  --single-transaction \
  --routines \
  --triggers \
  --add-drop-table \
  myapp_test > tests/snapshots/baseline.sql

gzip tests/snapshots/baseline.sql
```

`--single-transaction` creates a consistent snapshot for InnoDB without locking tables.

Restore the snapshot:

```bash
mysql -h 127.0.0.1 -u root -proot -e "
DROP DATABASE IF EXISTS myapp_test;
CREATE DATABASE myapp_test CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;
GRANT ALL ON myapp_test.* TO 'testuser'@'%';
"
gunzip -c tests/snapshots/baseline.sql.gz | \
  mysql -h 127.0.0.1 -u testuser -ptestpass myapp_test
```

## Method 2 - Schema-Only Snapshot

For test environments where data is generated by each test run, a schema-only snapshot reduces file size and restore time:

```bash
mysqldump \
  -h 127.0.0.1 \
  -u testuser \
  -ptestpass \
  --no-data \
  --routines \
  --triggers \
  myapp_test > tests/snapshots/schema.sql
```

This captures the exact table structure, indexes, stored procedures, and triggers but no rows.

## Method 3 - Docker Image Snapshot

Build a Docker image with MySQL pre-loaded with your test data. The image captures a complete MySQL data directory state:

```dockerfile
FROM mysql:8.0 AS builder

ENV MYSQL_ROOT_PASSWORD=testroot
ENV MYSQL_DATABASE=myapp_test

COPY tests/snapshots/baseline.sql /docker-entrypoint-initdb.d/01-baseline.sql

RUN ["/entrypoint.sh", "mysqld", "--initialize-insecure"]
```

Then in CI, start the pre-built image to skip migration and seeding time:

```yaml
services:
  mysql:
    image: myapp-test-db:latest
    environment:
      MYSQL_ROOT_PASSWORD: testroot
```

## Method 4 - LVM or Filesystem Snapshot

On dedicated test servers, LVM snapshots create an instant copy of the MySQL data directory:

```bash
# Create LVM snapshot of MySQL volume
lvcreate --size 5G --snapshot --name mysql_snap /dev/vg0/mysql_data

# Mount the snapshot
mount -o ro /dev/vg0/mysql_snap /mnt/mysql_snap

# Copy the data directory or use rsync to restore
```

This is the fastest snapshot method but requires LVM setup and cannot be used in typical CI containers.

## Versioning Snapshots

Store snapshot files in version control or a dedicated artifact store, versioned by migration number:

```bash
# Name snapshot by migration version
MIGRATION_VERSION=$(flyway info -outputType=json | jq -r '.schemaVersion')
mysqldump ... myapp_test > "tests/snapshots/v${MIGRATION_VERSION}-baseline.sql.gz"
```

When new migrations are applied, regenerate the snapshot:

```bash
make db:migrate db:seed db:snapshot
```

## Automating Snapshot Regeneration in CI

Add a CI job that regenerates snapshots when migrations change:

```yaml
- name: Check if migrations changed
  id: migrations
  run: git diff --name-only HEAD~1 | grep -q 'migrations/'

- name: Regenerate test snapshot
  if: steps.migrations.outputs.changes
  run: make db:snapshot
```

## Summary

Create MySQL test snapshots using `mysqldump --single-transaction` for portable SQL dumps, schema-only dumps for structure-only baselines, or Docker images with pre-loaded data for fastest container startup. Version snapshots by migration version and automate regeneration when schema changes land. Restoring from a snapshot is significantly faster than re-running all migrations and seeders for complex schemas.
