# How to Use the Dependency Lock File for Providers in OpenTofu

Author: [nawazdhandala](https://www.github.com/nawazdhandala)

Tags: OpenTofu, Terraform, Infrastructure as Code, Providers

Description: Learn how OpenTofu's dependency lock file works, why you should commit it to version control, and how to manage provider version upgrades with it.

## Introduction

The `.terraform.lock.hcl` file records the exact provider versions and checksums selected during `tofu init`. This lock file ensures that all team members and CI/CD pipelines use identical provider binaries, preventing "works on my machine" issues caused by provider version differences.

## What the Lock File Contains

```hcl
# .terraform.lock.hcl (auto-generated by tofu init)
provider "registry.opentofu.org/hashicorp/aws" {
  version     = "5.31.0"
  constraints = "~> 5.0"
  hashes = [
    "h1:abc123...",
    "zh:def456...",
  ]
}

provider "registry.opentofu.org/hashicorp/kubernetes" {
  version     = "2.25.0"
  constraints = ">= 2.20"
  hashes = [
    "h1:xyz789...",
  ]
}
```

The `hashes` field contains cryptographic checksums of the provider binaries for multiple platforms, ensuring the downloaded binary matches what was originally tested.

## Commit It to Version Control

```bash
# The lock file should always be committed
git add .terraform.lock.hcl
git commit -m "Lock provider versions after init"
```

Do not add it to `.gitignore`. The lock file is the mechanism that makes provider versions reproducible across your team.

## Upgrading Provider Versions

To upgrade providers within your constraints:

```bash
# Upgrade all providers to their latest allowed versions
tofu init -upgrade

# Review the diff before committing
git diff .terraform.lock.hcl
```

To upgrade a specific provider:

```bash
# Upgrade only the aws provider
tofu init -upgrade -plugin-dir=""
# (Note: targeted upgrade is done by adjusting version constraints)
```

## Adding New Platforms

The lock file records checksums for the current platform by default. To add checksums for other operating systems (e.g., Linux for CI when developing on macOS):

```bash
# Add checksums for Linux AMD64
tofu providers lock -platform=linux_amd64

# Add multiple platforms at once
tofu providers lock \
  -platform=linux_amd64 \
  -platform=darwin_arm64 \
  -platform=windows_amd64
```

## Lock File Conflicts

If two team members run `tofu init` simultaneously and select different patch versions, their lock files will conflict. Resolve by:

1. Choosing one version and updating `required_providers` to pin it exactly
2. Running `tofu init -upgrade` to regenerate with the agreed version
3. Committing the updated lock file

## When to Not Commit the Lock File

The only exception: module repositories that are consumed by other configurations. For modules, omit the lock file since the consuming configuration manages provider versions:

```
# .gitignore for a reusable module repository
.terraform/
.terraform.lock.hcl  # Omit for modules — the consumer manages this
```

For root configurations (deployable configurations), always commit the lock file.

## Conclusion

The `.terraform.lock.hcl` file is your insurance against unexpected provider behavior. Commit it to version control for all root configurations, use `tofu init -upgrade` to bump versions deliberately, and run `tofu providers lock -platform=...` to add checksums for all platforms used by your team and CI/CD system.
