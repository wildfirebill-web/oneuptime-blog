# How to Fix 'Maximum parse depth exceeded' in ClickHouse

Author: [nawazdhandala](https://www.github.com/nawazdhandala)

Tags: ClickHouse, Parsing, SQL Errors, Troubleshooting, Query Optimization

Description: Fix the ClickHouse 'Maximum parse depth exceeded' error by understanding its causes in deeply nested queries and adjusting parser limits.

---

## Understanding the Error

ClickHouse raises this error when it tries to parse a query whose abstract syntax tree (AST) exceeds the allowed nesting depth:

```text
DB::Exception: Maximum parse depth (2000) exceeded. Consider refactoring the query. (TOO_DEEP_RECURSION)
```

This most commonly happens with:
- Deeply nested `CASE WHEN ... THEN ... END` expressions
- Many layers of subqueries
- Auto-generated queries from ORMs or analytics tools
- Recursive CTEs that are deeply expanded

## Reproducing the Error

A simple example that can trigger deep parsing:

```sql
-- Deeply nested CASE expression (simplified, real cases have hundreds of levels)
SELECT
  CASE WHEN a = 1 THEN
    CASE WHEN b = 2 THEN
      CASE WHEN c = 3 THEN 'deep'
      ELSE 'c_other' END
    ELSE 'b_other' END
  ELSE 'a_other' END
FROM my_table;
```

Auto-generated queries from BI tools often produce hundreds of nesting levels.

## Fix 1 - Increase the Parse Depth Limit

The default max parse depth is 2000. You can increase it at the session or server level:

```sql
-- Temporarily raise the limit for this session
SET max_parser_depth = 5000;

-- Now run the deeply nested query
SELECT ... FROM my_table;
```

In `users.xml` for a persistent change:

```xml
<profiles>
  <default>
    <max_parser_depth>5000</max_parser_depth>
  </default>
</profiles>
```

## Fix 2 - Refactor Using WITH (CTEs)

Replace nested subqueries with CTEs to flatten the query structure:

```sql
-- Instead of nested subqueries:
SELECT *
FROM (
    SELECT *
    FROM (
        SELECT user_id, max(event_time) AS last_seen
        FROM events
        GROUP BY user_id
    ) AS t1
    WHERE last_seen > now() - INTERVAL 30 DAY
) AS t2
WHERE user_id IS NOT NULL;

-- Refactor as CTEs:
WITH
    user_last_seen AS (
        SELECT user_id, max(event_time) AS last_seen
        FROM events
        GROUP BY user_id
    ),
    recent_users AS (
        SELECT user_id, last_seen
        FROM user_last_seen
        WHERE last_seen > now() - INTERVAL 30 DAY
    )
SELECT *
FROM recent_users
WHERE user_id IS NOT NULL;
```

## Fix 3 - Replace Deep CASE Chains with Arrays

Convert long CASE expressions into array lookups:

```sql
-- Deep CASE approach (problematic):
SELECT
  CASE
    WHEN status = 'new' THEN 1
    WHEN status = 'active' THEN 2
    WHEN status = 'churned' THEN 3
    -- ... hundreds more
    ELSE 0
  END AS status_code
FROM users;

-- Array-based replacement:
SELECT
  indexOf(
    ['new', 'active', 'churned', 'paused', 'deleted'],
    status
  ) AS status_code
FROM users;
```

## Fix 4 - Use a Dictionary for Deep Lookups

For large mapping tables that inflate query depth when inlined:

```sql
-- Create a flat dictionary instead of nested CASE
CREATE DICTIONARY status_codes (
    status String,
    code UInt8
)
PRIMARY KEY status
SOURCE(CLICKHOUSE(TABLE 'status_map' DATABASE 'analytics'))
LAYOUT(HASHED())
LIFETIME(300);

-- Use it in the query
SELECT dictGet('status_codes', 'code', status) AS status_code
FROM users;
```

## Diagnosing the Depth

To understand how deep your query goes, use `EXPLAIN`:

```sql
EXPLAIN AST
SELECT ... FROM my_table WHERE ...;
```

Look for heavily nested `Function` or `Subquery` nodes in the output.

## Summary

The "Maximum parse depth exceeded" error occurs when ClickHouse's AST parser hits a recursion limit on deeply nested queries, often generated by ORMs or BI tools. You can resolve it by raising `max_parser_depth`, flattening nested subqueries into CTEs, replacing long CASE chains with array lookups, or offloading mappings to dictionaries. Refactoring is preferred over simply raising the limit, as deeply nested queries are also harder to optimize.
