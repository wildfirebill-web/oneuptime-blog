# How to Publish and Share Internal Modules for OpenTofu

Author: [nawazdhandala](https://www.github.com/nawazdhandala)

Tags: OpenTofu, Terraform, Modules, Module Registry, GitOps, Internal Sharing

Description: Learn how to publish and share internal OpenTofu modules using Git repositories, private module registries, and semantic versioning for team-wide consumption.

## Introduction

Sharing modules across teams requires a publishing mechanism and a clear versioning strategy. OpenTofu supports referencing modules from Git repositories, private registries, and S3/GCS buckets. Git tags with semantic versioning is the most widely adopted internal sharing approach.

## Git-Based Module Sharing

### Tagging Module Releases

```bash
# In your module repository

git tag -a v1.0.0 -m "Initial stable release of VPC module"
git tag -a v1.1.0 -m "Add support for IPv6 subnets"
git tag -a v2.0.0 -m "Breaking: rename subnet variables for consistency"
git push origin --tags
```

### Consuming Git-Tagged Modules

```hcl
# Reference a specific tag for production stability
module "vpc" {
  source = "git::https://github.com/your-org/tofu-modules.git//modules/vpc?ref=v2.1.0"

  name        = "prod-vpc"
  cidr_block  = "10.0.0.0/16"
  environment = "prod"
}

# Reference a branch for development (not recommended for prod)
module "vpc_dev" {
  source = "git::https://github.com/your-org/tofu-modules.git//modules/vpc?ref=feature/ipv6-support"
  # ...
}
```

## Automated Release with GitHub Actions

```yaml
# .github/workflows/release-module.yml
name: Release Module

on:
  push:
    tags:
      - 'v*'

jobs:
  validate:
    runs-on: ubuntu-latest
    strategy:
      matrix:
        module: [vpc, database, ecs-service, security-group]
    steps:
      - uses: actions/checkout@v4
      - uses: opentofu/setup-opentofu@v1

      - name: Validate module
        run: |
          cd modules/${{ matrix.module }}
          tofu init -backend=false
          tofu validate

      - name: Generate docs
        uses: terraform-docs/gh-actions@v1
        with:
          working-dir: modules/${{ matrix.module }}
          output-file: README.md

  release:
    needs: validate
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Create Release
        uses: actions/create-release@v1
        with:
          tag_name: ${{ github.ref }}
          release_name: Release ${{ github.ref }}
          body_path: CHANGELOG.md
```

## Private Registry Setup

For larger organizations, use a private Terraform/OpenTofu registry:

```hcl
# modules/vpc/versions.tf
terraform {
  # Declare what this module requires
  required_version = ">= 1.6.0"
  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = ">= 5.0"
    }
  }
}
```

```hcl
# Consuming from a private registry
module "vpc" {
  source  = "registry.your-org.com/networking/vpc/aws"
  version = "~> 2.0"
  # ...
}
```

## Conclusion

Git-based module sharing with semantic versioning and GitHub releases is the most accessible approach for most organizations. Automate validation and documentation generation in CI to ensure every released module is tested and documented. As your module library grows, consider moving to a private registry for better discoverability and access control.
