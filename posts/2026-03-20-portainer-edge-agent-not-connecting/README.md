# How to Fix Edge Agent Not Connecting to Portainer Server

Author: [nawazdhandala](https://www.github.com/nawazdhandala)

Tags: Portainer, Docker, Troubleshooting, Edge Agent, Edge Computing

Description: Diagnose and resolve Edge Agent connection failures to the Portainer server, including tunnel configuration, key issues, and network connectivity problems.

## Introduction

The Portainer Edge Agent is designed for environments where the Portainer server cannot directly reach the Docker host — the agent initiates an outbound connection (reverse tunnel) to the server. When this connection fails, edge devices appear offline and cannot be managed. This guide covers all common failure points.

## How Edge Agent Connectivity Works

Unlike the standard agent (where the server connects to the agent), the Edge Agent:
1. Initiates an outbound connection to the Portainer server on port 8000
2. Creates a reverse tunnel
3. Portainer uses this tunnel to send management commands

This means the edge device only needs outbound internet access — no inbound ports required.

## Step 1: Verify the Edge Agent Is Running

```bash
# On the edge device, check if the agent is running
docker ps | grep portainer-edge-agent

# Check logs
docker logs portainer-edge-agent --tail 50

# Look for connection messages
docker logs portainer-edge-agent 2>&1 | grep -i "tunnel\|connect\|server\|error"
```

## Step 2: Verify the Edge Key

The Edge Agent requires a key generated by the Portainer server:

```bash
# The key should have been provided when you created the Edge environment in Portainer
# Check the agent's configured key
docker inspect portainer-edge-agent | grep EDGE_KEY
```

If the key is missing or wrong:

1. In Portainer UI, go to **Environments** → find the Edge environment
2. Click the environment to view the deployment command
3. The command includes `EDGE_KEY=<your-key>`
4. Redeploy the agent with the correct key

## Step 3: Correct Edge Agent Deployment

```bash
# The standard Edge Agent deployment command (generated by Portainer)
docker run -d \
  -v /var/run/docker.sock:/var/run/docker.sock \
  -v /var/lib/docker/volumes:/var/lib/docker/volumes \
  -v /:/host \
  -v portainer_agent_data:/data \
  --restart always \
  -e EDGE=1 \
  -e EDGE_ID=<your-edge-id> \
  -e EDGE_KEY=<your-edge-key> \
  -e EDGE_INSECURE_POLL=1 \
  --name portainer-edge-agent \
  portainer/agent:latest
```

## Step 4: Verify Outbound Connectivity to Portainer

```bash
# Test outbound connection to Portainer server on port 8000
# (This is the tunnel port)
nc -zv portainer-server.yourdomain.com 8000

# Test HTTPS access to Portainer (port 9443)
curl -k https://portainer-server.yourdomain.com:9443/api/status

# If using Portainer Cloud or a custom tunnel address
nc -zv your-tunnel-host your-tunnel-port
```

## Step 5: Check Portainer Server Tunnel Port Configuration

The Portainer server must expose port 8000 for Edge Agent connections:

```bash
# Verify Portainer server has port 8000 exposed
docker inspect portainer | grep -A 20 '"PortBindings"'

# If 8000 is not exposed, recreate with it
docker stop portainer && docker rm portainer
docker run -d \
  -p 9000:9000 \
  -p 9443:9443 \
  -p 8000:8000 \   # Edge Agent tunnel port — REQUIRED
  --name portainer \
  --restart=unless-stopped \
  -v /var/run/docker.sock:/var/run/docker.sock \
  -v portainer_data:/data \
  portainer/portainer-ce:latest
```

## Step 6: Check Portainer Server Firewall

```bash
# On the Portainer server, ensure port 8000 is open
sudo ufw allow 8000/tcp
sudo ufw status

# Test from the edge device
nc -zv portainer-server-ip 8000
```

## Step 7: Configure Custom Tunnel Address

If Portainer is behind a reverse proxy or on a non-standard address:

```bash
# Set the tunnel server address when starting Portainer
docker run -d \
  -p 9000:9000 \
  -p 9443:9443 \
  -p 8000:8000 \
  --name portainer \
  -v /var/run/docker.sock:/var/run/docker.sock \
  -v portainer_data:/data \
  portainer/portainer-ce:latest \
  --tunnel-addr=0.0.0.0 \
  --tunnel-port=8000
```

On the Edge Agent side, the key encodes the server address, so regenerate the key after changing the server address.

## Step 8: Debug with Verbose Logging

```bash
# Run Edge Agent with debug logging
docker stop portainer-edge-agent && docker rm portainer-edge-agent

docker run -d \
  -v /var/run/docker.sock:/var/run/docker.sock \
  -v /var/lib/docker/volumes:/var/lib/docker/volumes \
  -v portainer_agent_data:/data \
  --restart always \
  -e EDGE=1 \
  -e EDGE_ID=<your-edge-id> \
  -e EDGE_KEY=<your-edge-key> \
  -e EDGE_INSECURE_POLL=1 \
  -e LOG_LEVEL=DEBUG \
  --name portainer-edge-agent \
  portainer/agent:latest

# Watch logs
docker logs -f portainer-edge-agent
```

## Step 9: Fix Clock Synchronization Issues

TLS connections (and edge agent authentication) require synchronized clocks:

```bash
# Check current time on edge device
date

# Sync with NTP
sudo timedatectl set-ntp true
sudo systemctl restart systemd-timesyncd

# Verify sync
timedatectl show | grep NTPSynchronized
```

## Step 10: Verify Edge Environment Status in Portainer

1. In Portainer, go to **Environments**
2. Find the Edge environment — check the **Last Check-in** timestamp
3. If stale (>5 minutes ago), the agent is not connecting
4. Click **Browse** → view the connection key details
5. Re-generate the key if needed and redeploy the agent

## Conclusion

Edge Agent connection failures are almost always caused by port 8000 not being accessible on the Portainer server, an incorrect or missing `EDGE_KEY`, or the tunnel server address being wrong in the agent configuration. Always verify that the Portainer server exposes port 8000, and that the edge device can reach it before debugging other factors.
