# How to Configure the MySQL Error Log

Author: [nawazdhandala](https://www.github.com/nawazdhandala)

Tags: MySQL, Error Log, Configuration, Logging, Diagnostic

Description: Learn how to configure the MySQL error log, set its location, control verbosity, and use it to diagnose startup failures and server errors.

---

The MySQL error log is the primary diagnostic tool for identifying server startup failures, crashes, and runtime warnings. It captures messages generated by `mysqld` during startup, shutdown, and normal operation. Knowing how to configure it properly is essential for any MySQL administrator.

## Default Error Log Location

By default, MySQL writes the error log to the data directory with the hostname as the filename:

```text
/var/lib/mysql/<hostname>.err
```

On systems using systemd with AppArmor or SELinux, the default path may be set by the distribution package.

## Configuring the Error Log in my.cnf

You configure the error log through the `[mysqld]` section of `my.cnf` (or `my.ini` on Windows):

```ini
[mysqld]
log_error = /var/log/mysql/error.log
```

After changing this setting, restart MySQL for it to take effect:

```bash
sudo systemctl restart mysql
```

Verify the path MySQL is using:

```sql
SHOW VARIABLES LIKE 'log_error';
```

## Controlling Log Verbosity

MySQL 8.0 introduced a more granular verbosity system using `log_error_verbosity`:

```ini
[mysqld]
log_error_verbosity = 3
```

The levels are:
- `1` - Errors only
- `2` - Errors and warnings
- `3` - Errors, warnings, and informational notes (default)

You can also change verbosity at runtime without a restart:

```sql
SET GLOBAL log_error_verbosity = 2;
```

## Using the Error Log Component System

MySQL 8.0 introduced log components for structured and filtered logging. You configure these with `log_error_services`:

```ini
[mysqld]
log_error_services = "log_filter_internal; log_sink_internal"
```

To add JSON-format logging alongside the traditional log:

```ini
[mysqld]
log_error_services = "log_filter_internal; log_sink_internal; log_sink_json"
```

Install the JSON sink component first:

```sql
INSTALL COMPONENT 'file://component_log_sink_json';
```

## Filtering Specific Error Codes

The `log_filter_dragnet` component lets you suppress or modify specific log events:

```sql
INSTALL COMPONENT 'file://component_log_filter_dragnet';
SET GLOBAL log_error_services = 'log_filter_dragnet; log_sink_internal';
SET GLOBAL dragnet.log_error_filter_rules = 'IF prio>=INFORMATION THEN drop. IF ERR_CODE == 1592 THEN drop.';
```

This suppresses informational messages and the unsafe statement warning (1592) for replication environments.

## Rotating the Error Log

On Linux, you can rotate the error log using `FLUSH ERROR LOGS` without restarting:

```sql
FLUSH ERROR LOGS;
```

To use logrotate, create `/etc/logrotate.d/mysql`:

```text
/var/log/mysql/error.log {
    daily
    missingok
    rotate 7
    compress
    delaycompress
    notifempty
    postrotate
        if test -x /usr/bin/mysqladmin && /usr/bin/mysqladmin ping &>/dev/null; then
            /usr/bin/mysqladmin flush-logs
        fi
    endscript
}
```

## Viewing Recent Error Log Entries

You can query recent error log entries directly from the `performance_schema`:

```sql
SELECT logged, thread_id, prio, error_code, subsystem, data
FROM performance_schema.error_log
ORDER BY logged DESC
LIMIT 20;
```

This is available in MySQL 8.0.22 and later and makes it easy to search for specific errors without needing shell access.

## Summary

The MySQL error log is your first resource when diagnosing problems. Configure it with a predictable path in `my.cnf`, set `log_error_verbosity` to match your environment's needs, and use log rotation to prevent unbounded growth. In MySQL 8.0, leverage the component-based logging system for structured output and fine-grained filtering.
