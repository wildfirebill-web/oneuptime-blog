# How to Troubleshoot Kubernetes Provider Connection Issues in OpenTofu

Author: [nawazdhandala](https://www.github.com/nawazdhandala)

Tags: OpenTofu, Troubleshooting, Kubernetes, EKS, Provider, Infrastructure as Code

Description: Learn how to diagnose and fix Kubernetes provider connection failures in OpenTofu including kubeconfig issues, EKS authentication errors, and provider configuration ordering problems.

## Introduction

The Kubernetes provider in OpenTofu connects to a cluster's API server to manage resources. Connection failures typically come from three sources: kubeconfig misconfiguration, authentication token expiration (especially with EKS), or provider configuration that runs before the cluster exists. Each has a distinct fix.

## Error: Unable to Connect to the Server

```text
Error: Get "https://abc123.gr7.us-east-1.eks.amazonaws.com/api/v1/namespaces":
dial tcp: lookup abc123.gr7.us-east-1.eks.amazonaws.com: no such host
```

```bash
# Test connectivity to the cluster endpoint

kubectl cluster-info

# Update kubeconfig for EKS
aws eks update-kubeconfig \
  --name my-cluster \
  --region us-east-1

# Test the connection
kubectl get nodes

# Check the kubeconfig being used
kubectl config view --minify
```

## Error: Unauthorized / Token Expired

```text
Error: Unauthorized
```

For EKS, tokens generated by `aws eks get-token` expire after 15 minutes. The Kubernetes provider uses exec-based authentication that re-fetches tokens automatically - but only if configured correctly.

```hcl
# Correct EKS provider configuration - uses exec for token refresh
provider "kubernetes" {
  host                   = aws_eks_cluster.main.endpoint
  cluster_ca_certificate = base64decode(aws_eks_cluster.main.certificate_authority[0].data)

  exec {
    api_version = "client.authentication.k8s.io/v1beta1"
    command     = "aws"
    args = [
      "eks", "get-token",
      "--cluster-name", aws_eks_cluster.main.name,
      "--region", "us-east-1"
    ]
  }
}
```

Avoid using static tokens - they expire and cannot be refreshed automatically.

## Error: Provider Configured Before Cluster Exists

This is the most common Kubernetes provider issue in OpenTofu.

```text
Error: Post "https://<null>/api/v1/namespaces":
dial tcp: address <null>: missing port in address
```

This happens when the Kubernetes provider is configured with `aws_eks_cluster.main.endpoint` but OpenTofu tries to initialize the provider before the cluster is created.

```hcl
# PROBLEMATIC: Provider configuration depends on a resource
# that might not exist yet during planning
provider "kubernetes" {
  host = aws_eks_cluster.main.endpoint  # null during first plan
}
```

The solution is to split the configuration into separate state files or use the `depends_on` on provider configuration (not supported) or structure the code properly.

```hcl
# Solution 1: Use data sources in a separate root module
# cluster/main.tf - creates the EKS cluster
# k8s-resources/main.tf - uses data source to get cluster info

# In k8s-resources/main.tf:
data "aws_eks_cluster" "main" {
  name = "my-cluster"
}

data "aws_eks_cluster_auth" "main" {
  name = "my-cluster"
}

provider "kubernetes" {
  host                   = data.aws_eks_cluster.main.endpoint
  cluster_ca_certificate = base64decode(data.aws_eks_cluster.main.certificate_authority[0].data)
  token                  = data.aws_eks_cluster_auth.main.token
}
```

## Error: aws-auth ConfigMap Access Denied

After creating an EKS cluster, only the creator IAM entity can access it by default.

```bash
# Check who created the cluster (has implicit admin)
aws eks describe-cluster \
  --name my-cluster \
  --query 'cluster.roleArn'

# View current aws-auth ConfigMap
kubectl get configmap aws-auth -n kube-system -o yaml
```

```hcl
# Grant access to additional IAM roles via aws-auth
resource "kubernetes_config_map_v1_data" "aws_auth" {
  metadata {
    name      = "aws-auth"
    namespace = "kube-system"
  }

  data = {
    mapRoles = yamlencode([
      {
        rolearn  = aws_iam_role.node_group.arn
        username = "system:node:{{EC2PrivateDNSName}}"
        groups   = ["system:bootstrappers", "system:nodes"]
      },
      {
        rolearn  = "arn:aws:iam::123456789012:role/OpenTofuCIRole"
        username = "opentofu-ci"
        groups   = ["system:masters"]
      }
    ])
  }

  force = true
}
```

## Debugging Provider Connection

```bash
# Enable Kubernetes provider debug logging
TF_LOG=DEBUG tofu plan 2>&1 | grep -i "kubernetes\|k8s\|dial\|tls\|auth"

# Test the exact token generation command
aws eks get-token --cluster-name my-cluster --region us-east-1

# Verify the token works
kubectl --token=$(aws eks get-token --cluster-name my-cluster \
  --region us-east-1 --query 'status.token' --output text) \
  get nodes
```

## Multiple Kubernetes Providers

When managing multiple clusters, use provider aliases.

```hcl
provider "kubernetes" {
  alias = "prod"
  host  = data.aws_eks_cluster.prod.endpoint
  # ...exec block...
}

provider "kubernetes" {
  alias = "staging"
  host  = data.aws_eks_cluster.staging.endpoint
  # ...exec block...
}

resource "kubernetes_namespace" "monitoring" {
  provider = kubernetes.prod
  metadata { name = "monitoring" }
}
```

## Summary

Kubernetes provider connection issues follow a pattern: if the endpoint is `<null>`, the cluster doesn't exist yet - split into two root modules and use a data source. If authentication fails with 401/Unauthorized on EKS, switch from static tokens to the `exec` block with `aws eks get-token`. If `aws-auth` access is denied, ensure the IAM role running OpenTofu is listed in the ConfigMap. Enable `TF_LOG=DEBUG` to see the exact HTTP requests being made to the API server.
