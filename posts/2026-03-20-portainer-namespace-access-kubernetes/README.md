# How to Configure Namespace-Level Access in Portainer for Kubernetes (2)

Author: [nawazdhandala](https://www.github.com/nawazdhandala)

Tags: Portainer, Kubernetes, Namespace, Access Control, RBAC

Description: Configure Portainer to restrict Kubernetes users to specific namespaces, enabling multi-team Kubernetes cluster management with isolation.

## Introduction

In Kubernetes environments, namespace-level access control allows multiple teams to share a single cluster while maintaining isolation. Portainer provides namespace-based access control that maps to Kubernetes RBAC, allowing you to give teams access to specific namespaces without requiring Kubernetes RBAC expertise.

## Prerequisites

- Portainer connected to a Kubernetes environment
- Kubernetes environment with multiple namespaces (or plans to create them)
- Portainer teams configured for each namespace access group

## Step 1: Enable Namespace Access Management

1. Go to **Environments** → select the Kubernetes environment
2. Click the **Settings** tab
3. Enable **Namespace access management**
4. Save

This enables namespace-level access control in the Portainer UI.

## Step 2: Create Namespaces

```bash
# Create namespaces via Portainer or directly via kubectl

kubectl create namespace backend-team
kubectl create namespace frontend-team
kubectl create namespace data-team
kubectl create namespace shared-services

# Or in Portainer:
# Go to Environments → [K8s Env] → Namespaces → Add namespace
```

Via Portainer API:
```bash
TOKEN=$(curl -s -X POST \
  https://portainer.example.com/api/auth \
  -H "Content-Type: application/json" \
  -d '{"username":"admin","password":"adminpassword"}' \
  | python3 -c "import sys,json; print(json.load(sys.stdin)['jwt'])")

ENDPOINT_ID=5  # Your Kubernetes environment ID

# Create namespace via Portainer API
curl -X POST \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  "https://portainer.example.com/api/endpoints/${ENDPOINT_ID}/kubernetes/namespaces" \
  -d '{
    "Name": "backend-team",
    "Owner": "",
    "ResourceQuota": {
      "enabled": true,
      "cpu": "2",
      "memory": "4Gi"
    }
  }'
```

## Step 3: Configure Namespace Access Policies

### Via UI

1. Go to **Environments** → select K8s environment
2. Click **Namespaces**
3. Click on a namespace (e.g., `backend-team`)
4. Click **Manage access** or view the **Access** tab
5. Add the team with the appropriate role

### Via API

```bash
ENDPOINT_ID=5
NAMESPACE="backend-team"
TEAM_ID=2  # Backend team

curl -X PUT \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  "https://portainer.example.com/api/endpoints/${ENDPOINT_ID}/kubernetes/namespaces/${NAMESPACE}/access" \
  -d "{
    \"TeamAccessPolicies\": {
      \"${TEAM_ID}\": {
        \"RoleId\": 2
      }
    },
    \"UserAccessPolicies\": {}
  }"
```

## Step 4: Test Namespace Isolation

Log in as a team member and verify they only see their namespace:

```bash
# Login as backend team member
BACKEND_TOKEN=$(curl -s -X POST \
  https://portainer.example.com/api/auth \
  -H "Content-Type: application/json" \
  -d '{"username":"backend-user","password":"password"}' \
  | python3 -c "import sys,json; print(json.load(sys.stdin)['jwt'])")

# List accessible namespaces
curl -s \
  -H "Authorization: Bearer $BACKEND_TOKEN" \
  "https://portainer.example.com/api/endpoints/${ENDPOINT_ID}/kubernetes/namespaces" \
  | python3 -c "import sys,json; [print(ns['Name']) for ns in json.load(sys.stdin)]"

# Should only show: backend-team (and default, kube-system as read-only)
```

Resource Quotas per Namespace

Set resource quotas to prevent one team from consuming all cluster resources:

```yaml
# Via kubectl or Portainer's Namespace Settings
apiVersion: v1
kind: ResourceQuota
metadata:
  name: backend-team-quota
  namespace: backend-team
spec:
  hard:
    requests.cpu: "4"
    requests.memory: 8Gi
    limits.cpu: "8"
    limits.memory: 16Gi
    pods: "50"
    services: "20"
    persistentvolumeclaims: "10"
```

In Portainer:
1. Go to Namespaces → backend-team → **Resource Quota**
2. Enable quota and set limits
3. Save

## Kubernetes RBAC Generated by Portainer

When Portainer assigns namespace access, it creates Kubernetes RBAC resources:

```yaml
# Generated ClusterRole for Standard User in namespace
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name: portainer-standard-user
  namespace: backend-team
rules:
  - apiGroups: ["", "apps", "batch"]
    resources: ["*"]
    verbs: ["get", "list", "watch", "create", "update", "patch", "delete"]

---
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: portainer-team-2-binding
  namespace: backend-team
subjects:
  - kind: Group
    name: "portainer-team-2"
roleRef:
  kind: Role
  name: portainer-standard-user
  apiGroup: rbac.authorization.k8s.io
```

## Conclusion

Namespace-level access control in Portainer enables multi-team Kubernetes cluster sharing with proper isolation. Teams can self-service their namespace deployments without interfering with each other, while the platform team maintains cluster-level control. Combined with resource quotas, namespace isolation provides both access control and resource fairness across teams.
