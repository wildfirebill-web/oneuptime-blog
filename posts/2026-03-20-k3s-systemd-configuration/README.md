# How to Configure K3s with Systemd

Author: [nawazdhandala](https://www.github.com/nawazdhandala)

Tags: K3s, Kubernetes, Systemd, Linux, Service Management, DevOps

Description: Learn how to configure and manage K3s systemd service units, including resource limits, environment variables, and service dependencies.

## Introduction

K3s installs as a systemd service on Linux systems. Understanding and customizing the systemd service configuration is essential for production deployments — from setting resource limits to configuring startup dependencies and environment variables. This guide covers comprehensive systemd configuration for K3s.

## Understanding the K3s Systemd Service

K3s installs two systemd services:
- **`k3s.service`**: The K3s server service
- **`k3s-agent.service`**: The K3s agent service

These are located at `/etc/systemd/system/k3s.service` and `/etc/systemd/system/k3s-agent.service`.

## Step 1: View the Default Service Files

```bash
# View the K3s server service file
systemctl cat k3s

# View the K3s agent service file
systemctl cat k3s-agent

# View the environment file
cat /etc/systemd/system/k3s.service.env
```

The default K3s service file:

```ini
# /etc/systemd/system/k3s.service (auto-generated by installer)
[Unit]
Description=Lightweight Kubernetes
Documentation=https://k3s.io
Wants=network-online.target
After=network-online.target

[Install]
WantedBy=multi-user.target

[Service]
Type=notify
EnvironmentFile=-/etc/default/%N
EnvironmentFile=-/etc/sysconfig/%N
EnvironmentFile=-/etc/systemd/system/%N.service.env
KillMode=process
Delegate=yes
# Having non-zero Limit*s causes performance problems due to accounting overhead
# in the kernel. We recommend using cgroups to do container-level resource
# temporary workaround for memory.high and memory.max
LimitNOFILE=1048576
LimitNPROC=infinity
LimitCORE=infinity
TasksMax=infinity
TimeoutStartSec=0
Restart=always
RestartSec=5s
ExecStartPre=/bin/sh -xc '! /usr/bin/systemctl is-enabled --quiet nm-cloud-setup.service'
ExecStartPre=-/sbin/modprobe br_netfilter
ExecStartPre=-/sbin/modprobe overlay
ExecStart=/usr/local/bin/k3s \
    server \
```

## Step 2: Customize Using Drop-in Files

Use systemd drop-in files instead of modifying the main service file (survives K3s upgrades):

```bash
# Create drop-in directory
mkdir -p /etc/systemd/system/k3s.service.d/

# Create a drop-in file
cat > /etc/systemd/system/k3s.service.d/override.conf << 'EOF'
[Service]
# Override environment variables
Environment="K3S_KUBECONFIG_MODE=644"

# Set resource limits
CPUQuota=80%
MemoryMax=4G
MemoryHigh=3G

# Change restart behavior
RestartSec=10s
StartLimitIntervalSec=60
StartLimitBurst=3

# Log to syslog with identifier
SyslogIdentifier=k3s-server

# Set working directory
WorkingDirectory=/var/lib/rancher/k3s
EOF

# Reload and restart
systemctl daemon-reload
systemctl restart k3s
```

## Step 3: Configure Environment Variables

```bash
# K3s reads from the environment file automatically
cat > /etc/systemd/system/k3s.service.env << 'EOF'
# K3s server token
K3S_TOKEN=your-secure-token-here

# Kubeconfig permissions (makes it readable by non-root)
K3S_KUBECONFIG_MODE=644

# For K3s agent
K3S_URL=https://server-ip:6443
K3S_TOKEN=your-secure-token-here
EOF

# Protect the env file (contains secrets)
chmod 600 /etc/systemd/system/k3s.service.env

# Reload and restart
systemctl daemon-reload
systemctl restart k3s
```

## Step 4: Add Service Dependencies

Configure K3s to wait for specific services (like NFS mounts):

```ini
# /etc/systemd/system/k3s.service.d/dependencies.conf
[Unit]
# Wait for NFS mounts before starting
After=network-online.target nfs-client.target remote-fs.target
Wants=nfs-client.target remote-fs.target

# Ensure K3s starts after database (if using external datastore)
After=postgresql.service
Requires=postgresql.service
```

## Step 5: Configure Resource Limits for Production

```ini
# /etc/systemd/system/k3s.service.d/limits.conf
[Service]
# CPU limits - allow K3s to use up to 4 CPU cores
CPUQuota=400%
CPUWeight=100

# Memory limits
MemoryMax=8G
MemoryHigh=7G
MemorySwapMax=0  # No swap for K3s

# File descriptor limits
LimitNOFILE=1048576
LimitNPROC=infinity

# Prevent core dumps
LimitCORE=0

# IO limits (for storage-intensive workloads)
IOWeight=100
IOReadBandwidthMax=/dev/sda 500M
IOWriteBandwidthMax=/dev/sda 200M
```

## Step 6: Configure Logging

```ini
# /etc/systemd/system/k3s.service.d/logging.conf
[Service]
# Send logs to both journal and syslog
StandardOutput=journal+console
StandardError=journal+console
SyslogIdentifier=k3s

# Set log level via environment
Environment="K3S_LOG_LEVEL=info"
```

Configure journal log retention for K3s:

```bash
# Limit K3s journal size
mkdir -p /etc/systemd/journald.conf.d/

cat > /etc/systemd/journald.conf.d/k3s.conf << 'EOF'
[Journal]
# Keep 500MB of journal logs
SystemMaxUse=500M
# Max size per file
SystemMaxFileSize=100M
# Keep logs for 7 days
MaxRetentionSec=7day
EOF

# Restart journald
systemctl restart systemd-journald
```

## Step 7: Configure Watchdog for Enhanced Reliability

```ini
# /etc/systemd/system/k3s.service.d/watchdog.conf
[Service]
# Enable systemd watchdog (K3s notifies watchdog via sd_notify)
WatchdogSec=60s
# Restart K3s if it stops sending watchdog pings
Restart=always
RestartSec=5s

# Limit restart attempts
StartLimitIntervalSec=300
StartLimitBurst=5

# Action when start limit is hit
StartLimitAction=reboot-force
```

## Step 8: Systemd Timer for Maintenance Tasks

Create a systemd timer for K3s maintenance:

```ini
# /etc/systemd/system/k3s-maintenance.service
[Unit]
Description=K3s Maintenance Tasks
After=k3s.service

[Service]
Type=oneshot
User=root
ExecStart=/usr/local/bin/k3s-maintenance.sh
```

```ini
# /etc/systemd/system/k3s-maintenance.timer
[Unit]
Description=Run K3s maintenance tasks daily
Requires=k3s-maintenance.service

[Timer]
# Run daily at 2 AM
OnCalendar=*-*-* 02:00:00
Persistent=true

[Install]
WantedBy=timers.target
```

```bash
# Create maintenance script
cat > /usr/local/bin/k3s-maintenance.sh << 'EOF'
#!/bin/bash
# Daily K3s maintenance

# Clean up unused container images
k3s crictl rmi --prune

# Clean up stopped containers
k3s crictl rm $(k3s crictl ps -a -q 2>/dev/null) 2>/dev/null || true

# Take etcd snapshot if using embedded etcd
k3s etcd-snapshot save --name "daily-$(date +%Y%m%d)" 2>/dev/null || true

# Prune old snapshots (keep last 7)
k3s etcd-snapshot list 2>/dev/null | \
  sort -t' ' -k2 -r | \
  tail -n +8 | \
  awk '{print $1}' | \
  xargs -I{} k3s etcd-snapshot delete {} 2>/dev/null || true

echo "K3s maintenance complete: $(date)"
EOF
chmod +x /usr/local/bin/k3s-maintenance.sh

# Enable the timer
systemctl enable k3s-maintenance.timer
systemctl start k3s-maintenance.timer
```

## Step 9: Common Systemd Commands for K3s

```bash
# Start K3s
systemctl start k3s

# Stop K3s
systemctl stop k3s

# Restart K3s
systemctl restart k3s

# Reload K3s configuration (graceful, no restart)
systemctl reload k3s  # Not always supported

# Check status
systemctl status k3s

# View logs
journalctl -u k3s -f                    # Follow live
journalctl -u k3s --since "1 hour ago"  # Last hour
journalctl -u k3s -n 100               # Last 100 lines
journalctl -u k3s --until "2024-03-15 10:00:00"

# Enable/disable auto-start
systemctl enable k3s
systemctl disable k3s

# Check all K3s-related services
systemctl list-units | grep k3s
```

## Conclusion

Proper systemd configuration is fundamental to reliable K3s operation in production. Use drop-in files for customizations to ensure they survive K3s upgrades. Key configurations include resource limits to prevent K3s from consuming all system resources, service dependencies for proper startup ordering, and watchdog configuration for automatic recovery. For production environments, always test systemd configuration changes in a staging environment before applying to production clusters.
