# How to Store UUID Values in MySQL Efficiently

Author: [nawazdhandala](https://www.github.com/nawazdhandala)

Tags: MySQL, UUID, Primary Key, Performance, Database

Description: Learn how to store UUID values efficiently in MySQL using binary storage and helper functions to reduce space and improve index performance.

---

## Why UUID Storage Matters

UUIDs (Universally Unique Identifiers) are popular primary keys because they can be generated client-side without a database round-trip. However, storing a UUID as a `CHAR(36)` or `VARCHAR(36)` string wastes space and degrades index performance. A 36-character UUID string takes 36 bytes; a `BINARY(16)` column stores the same data in 16 bytes.

## Option 1 - Store as CHAR(36) - Simple but Inefficient

```sql
CREATE TABLE users (
  id CHAR(36) NOT NULL DEFAULT (UUID()),
  name VARCHAR(100) NOT NULL,
  PRIMARY KEY (id)
);
```

This is easy to use but has poor index performance at scale due to the large key size and random ordering causing page splits.

## Option 2 - Store as BINARY(16) with UUID_TO_BIN

MySQL 8.0 introduced `UUID_TO_BIN()` and `BIN_TO_UUID()` to convert between string and binary representations:

```sql
CREATE TABLE users (
  id BINARY(16) NOT NULL DEFAULT (UUID_TO_BIN(UUID())),
  name VARCHAR(100) NOT NULL,
  PRIMARY KEY (id)
);
```

Insert with explicit conversion:

```sql
INSERT INTO users (id, name) VALUES (UUID_TO_BIN(UUID()), 'Alice');
```

Query and display as readable string:

```sql
SELECT BIN_TO_UUID(id) AS id, name FROM users;
```

## Using the swap_flag for Sequential UUIDs

The `UUID_TO_BIN()` function accepts an optional second argument. When set to 1, it rearranges the time components of a UUID v1 to make the bytes sequential, reducing B-tree page splits:

```sql
INSERT INTO users (id, name)
VALUES (UUID_TO_BIN(UUID(), 1), 'Bob');

SELECT BIN_TO_UUID(id, 1) AS id, name FROM users;
```

This swap flag is only meaningful for UUID v1 values generated by MySQL's `UUID()` function.

## Comparing Storage Sizes

```sql
-- Check storage footprint
SELECT
  CHAR_LENGTH(UUID()) AS string_length,         -- 36 chars
  LENGTH(UUID_TO_BIN(UUID())) AS binary_length;  -- 16 bytes
```

For a table with 10 million rows, this difference saves roughly 200 MB of primary key data alone.

## Using UUID as a Secondary Key

If you want an auto-increment integer as the primary key with UUID as a unique secondary key:

```sql
CREATE TABLE sessions (
  id INT AUTO_INCREMENT PRIMARY KEY,
  session_uuid BINARY(16) NOT NULL DEFAULT (UUID_TO_BIN(UUID())),
  user_id INT NOT NULL,
  UNIQUE KEY uq_session (session_uuid)
);
```

## Summary

Storing UUIDs as `BINARY(16)` instead of `CHAR(36)` cuts storage size by more than half and improves index efficiency. MySQL 8.0's `UUID_TO_BIN()` and `BIN_TO_UUID()` functions make this conversion straightforward. For UUID v1 values, use the swap flag to make keys sequential and reduce index fragmentation.
