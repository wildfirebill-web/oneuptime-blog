# How to Use SHOW ERRORS in MySQL

Author: [nawazdhandala](https://www.github.com/nawazdhandala)

Tags: MySQL, Error, Debugging

Description: Learn how to use SHOW ERRORS in MySQL to retrieve error messages from the most recently executed statement for debugging and diagnosis.

---

## What Is SHOW ERRORS

`SHOW ERRORS` displays error messages generated by the most recently executed statement in the current session. MySQL maintains an internal diagnostics area that stores errors, warnings, and notes from the last executed command. `SHOW ERRORS` retrieves only the error-level messages from this area.

```sql
SHOW ERRORS;
SHOW ERRORS LIMIT n;
SHOW ERRORS LIMIT offset, count;
```

## Difference Between SHOW ERRORS and SHOW WARNINGS

MySQL uses three severity levels:

- **Error**: Statement failed, operation was not performed
- **Warning**: Statement succeeded but with potential data loss or unexpected behavior
- **Note**: Informational message, no data impact

```sql
-- Shows only errors
SHOW ERRORS;

-- Shows errors, warnings, and notes
SHOW WARNINGS;
```

After a successful statement with no errors, `SHOW ERRORS` returns an empty result set.

## Output Columns

```text
+-------+------+---------------------------------------------+
| Level | Code | Message                                     |
+-------+------+---------------------------------------------+
| Error | 1062 | Duplicate entry '5' for key 'PRIMARY'       |
+-------+------+---------------------------------------------+
```

- **Level**: Always `Error` for results from `SHOW ERRORS`
- **Code**: MySQL error number (e.g., 1062 for duplicate key)
- **Message**: Human-readable error description

## Triggering an Error for Demonstration

```sql
-- Try to insert a duplicate primary key
INSERT INTO users (id, name) VALUES (1, 'Alice');
INSERT INTO users (id, name) VALUES (1, 'Duplicate');  -- Error

-- Now view the error
SHOW ERRORS;
```

```text
+-------+------+----------------------------------------------+
| Level | Code | Message                                      |
+-------+------+----------------------------------------------+
| Error | 1062 | Duplicate entry '1' for key 'users.PRIMARY' |
+-------+------+----------------------------------------------+
```

## Using GET DIAGNOSTICS

MySQL 5.6+ supports the SQL standard `GET DIAGNOSTICS` which provides more detailed error information and can be used inside stored procedures:

```sql
DECLARE v_code INT;
DECLARE v_msg TEXT;

GET DIAGNOSTICS CONDITION 1
  v_code = MYSQL_ERRNO,
  v_msg = MESSAGE_TEXT;
```

## Using @@error_count

You can check the number of errors from the last statement without fetching them:

```sql
SELECT @@error_count;
```

Combined with `@@warning_count`:

```sql
-- Run a statement
INSERT INTO orders (id, product_id) VALUES (NULL, 999);

-- Check counts
SELECT @@error_count AS errors, @@warning_count AS warnings;
```

## Error Handling in Stored Procedures

In stored procedures, use `DECLARE ... HANDLER` to handle errors gracefully:

```sql
DELIMITER $$
CREATE PROCEDURE safe_insert(IN p_id INT, IN p_name VARCHAR(100))
BEGIN
  DECLARE EXIT HANDLER FOR 1062
  BEGIN
    SHOW ERRORS;
    SELECT 'Duplicate key detected - insert skipped' AS result;
  END;

  INSERT INTO users (id, name) VALUES (p_id, p_name);
  SELECT 'Insert successful' AS result;
END$$
DELIMITER ;
```

## Limiting Error Output

If multiple errors occurred (possible with multi-row statements), limit the output:

```sql
-- Show only the first 3 errors
SHOW ERRORS LIMIT 3;

-- Show errors starting from the second (offset 1, count 5)
SHOW ERRORS LIMIT 1, 5;
```

## Combining with SHOW WARNINGS for Full Context

In practice, `SHOW WARNINGS` is more commonly used because it includes errors along with warnings and notes:

```sql
-- Run a problematic statement
LOAD DATA INFILE '/tmp/data.csv' INTO TABLE products;

-- See all issues (errors + warnings + notes)
SHOW WARNINGS;

-- See just errors
SHOW ERRORS;
```

## Summary

`SHOW ERRORS` displays error messages from the most recently executed MySQL statement. Use it for quick diagnostics after a failed operation. For more complete diagnostics including warnings and notes, prefer `SHOW WARNINGS`. In stored procedures, use `GET DIAGNOSTICS` or `DECLARE ... HANDLER` for programmatic error handling. Check `@@error_count` to quickly determine whether any errors occurred.
