# How to Implement Idempotent Payment Processing with MongoDB

Author: [nawazdhandala](https://www.github.com/nawazdhandala)

Tags: MongoDB, Payment, Idempotency, Transaction, Reliability

Description: Implement idempotent payment processing with MongoDB using unique idempotency keys to prevent duplicate charges from retries, network failures, and client timeouts.

---

## Why Idempotency Matters in Payments

Network failures and timeouts create ambiguity: did the payment go through or not? Without idempotency, a client that retries after a timeout might charge the customer twice. Idempotent APIs ensure that repeating the same request produces the same result without side effects.

## Idempotency Key Strategy

Assign a unique idempotency key to each payment attempt. The key should be:
- Generated by the client before initiating the payment
- Derived from the business context (order ID + attempt number)
- Stored with the transaction record

```javascript
function generateIdempotencyKey(orderId, attemptNumber = 1) {
  return `${orderId}-attempt-${attemptNumber}`;
}
```

## Database Schema for Idempotent Payments

```javascript
// Unique index on idempotency key prevents duplicate processing
db.transactions.createIndex(
  { idempotencyKey: 1 },
  { unique: true }
)

// Transaction document structure
{
  _id: "txn_abc123",
  idempotencyKey: "ord-001-attempt-1",
  orderId: "ord-001",
  status: "processing",
  amount: 5000,
  currency: "USD",
  createdAt: ISODate("2026-03-31T10:00:00Z"),
  response: null   // Populated after processor call
}
```

## Implementing the Idempotent Payment Handler

```javascript
async function processPayment(orderId, amount, paymentMethod) {
  const idempotencyKey = generateIdempotencyKey(orderId);

  // Step 1: Try to create the transaction record atomically
  let transaction;
  try {
    transaction = await db.collection("transactions").findOneAndUpdate(
      { idempotencyKey },
      {
        $setOnInsert: {
          _id: generateId(),
          idempotencyKey,
          orderId,
          amount,
          currency: "USD",
          paymentMethod,
          status: "pending",
          createdAt: new Date()
        }
      },
      {
        upsert: true,
        returnDocument: "after"
      }
    );
  } catch (err) {
    if (err.code === 11000) {
      // Race condition - another request just created this transaction
      transaction = await db.collection("transactions").findOne({ idempotencyKey });
    } else {
      throw err;
    }
  }

  // Step 2: If already succeeded or failed, return existing result
  if (transaction.status === "succeeded") {
    return { duplicate: true, transaction };
  }

  if (transaction.status === "failed") {
    return { duplicate: true, transaction, error: transaction.failureReason };
  }

  // Step 3: Call the payment processor
  let processorResult;
  try {
    processorResult = await stripe.charges.create({
      amount,
      currency: "usd",
      source: paymentMethod,
      idempotency_key: idempotencyKey  // Pass key to processor too
    });

    await db.collection("transactions").updateOne(
      { idempotencyKey, status: "pending" },
      {
        $set: {
          status: "succeeded",
          processorId: processorResult.id,
          processedAt: new Date(),
          response: processorResult
        }
      }
    );

    return { duplicate: false, transaction: { ...transaction, status: "succeeded" } };
  } catch (err) {
    await db.collection("transactions").updateOne(
      { idempotencyKey },
      {
        $set: {
          status: "failed",
          failureReason: err.message,
          failedAt: new Date()
        }
      }
    );
    throw err;
  }
}
```

## Handling Concurrent Retries

When multiple retries arrive simultaneously (e.g., from a message queue retry), `findOneAndUpdate` with `$setOnInsert` ensures only one succeeds at creating the record:

```javascript
// All concurrent calls with the same idempotencyKey get the same result
const [r1, r2, r3] = await Promise.allSettled([
  processPayment("ord-001", 5000, "pm_xyz"),
  processPayment("ord-001", 5000, "pm_xyz"),
  processPayment("ord-001", 5000, "pm_xyz")
]);
// Only one actually processes the payment; others return the existing result
```

## Expiring Old Idempotency Keys

Use a TTL index to clean up old keys (24-hour window is common):

```javascript
db.transactions.createIndex(
  { createdAt: 1 },
  { expireAfterSeconds: 86400 }  // 24 hours
)
```

## Summary

Idempotent payment processing with MongoDB uses a unique index on an idempotency key to prevent duplicate transactions. The `findOneAndUpdate` with `$setOnInsert` atomically creates the transaction record only if it does not exist, allowing concurrent retries to safely converge on the same result. Pass the idempotency key to the payment processor as well to prevent duplicate charges at the processor level. Return cached results for any repeat request with the same key, whether it succeeded, failed, or is still processing.
