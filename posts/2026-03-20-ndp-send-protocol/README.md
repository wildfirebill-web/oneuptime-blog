# How to Understand Secure Neighbor Discovery Protocol (SEND)

Author: [nawazdhandala](https://www.github.com/nawazdhandala)

Tags: SEND, NDP Security, IPv6, Cryptographic Addresses, RFC 3971

Description: Understand how SEND (Secure Neighbor Discovery) provides cryptographic authentication for NDP messages, preventing rogue RA and NA attacks at the protocol level.

## Introduction

SEND (Secure Neighbor Discovery, RFC 3971) adds cryptographic authentication to IPv6 NDP messages to prevent rogue Router Advertisement and Neighbor Advertisement attacks. SEND uses Cryptographically Generated Addresses (CGA, RFC 3972) and RSA digital signatures to ensure NDP messages can only be generated by the legitimate owner of an IPv6 address. SEND is the protocol-level security solution, while RA Guard is the simpler switch-level alternative.

## SEND Architecture

```
SEND Security Model:

Without SEND:
  Any host on the link can send:
    - Forged Router Advertisement (become a rogue gateway)
    - Forged Neighbor Advertisement (poison neighbor cache)
  These attacks require only link-local access

With SEND:
  Each NDP message includes:
    - CGA option: Proves sender owns the IPv6 address
    - RSA Signature option: Proves message wasn't tampered with
    - Timestamp option: Prevents replay attacks
    - Nonce option: Prevents reflection attacks

  Attackers cannot forge these because:
    - CGA address is bound to a public key via hash
    - Signature requires the private key

SEND Requirements:
  - All NDP-speaking nodes need RSA key pairs
  - Routers need router authorization certificates
  - PKI infrastructure needed for router authorization
  - Computationally expensive (RSA operations per NDP message)
```

## How CGA Works

```
Cryptographically Generated Addresses (RFC 3972):

1. Node generates an RSA key pair
2. Forms CGA by:
   a. Hash = SHA1(public_key + subnet_prefix + collision_count + modifier)
   b. Set CGA interface identifier = leftmost 64 bits of Hash
   c. Set bits u and g in interface identifier appropriately
3. The IPv6 address = subnet_prefix + CGA_interface_identifier

To verify CGA ownership:
  1. Receive NDP message with CGA and RSA Signature options
  2. Re-compute Hash(public_key + subnet_prefix + ...)
  3. Verify hash matches the low bits of the source IPv6 address
  4. Verify RSA signature using the public key from CGA option
  5. If both match: sender is legitimate owner of the address

Attack resistance:
  Forging a CGA address requires:
  - Finding a collision in SHA1 (computationally infeasible)
  - Or breaking RSA (computationally infeasible)
```

## SEND Message Extensions

```
Additional NDP options used by SEND:

CGA Option (Type 11):
  Contains the public key and CGA Parameters
  Used to verify address ownership

RSA Signature Option (Type 12):
  RSA digital signature over the NDP message + CGA info
  Signs the entire message to prevent tampering

Timestamp Option (Type 13):
  Current time to prevent replay attacks
  Receivers reject messages with timestamps too far in the past/future

Nonce Option (Type 14):
  Random value used to prevent reflection attacks
  NS includes nonce; corresponding NA must echo it back
  (without SEND: an attacker could replay a captured NA)
```

## SEND Deployment Considerations

```
SEND Challenges:

1. Computational overhead:
   RSA signatures are CPU-intensive
   High-rate NDP traffic (ND floods) requires many RSA verifications

2. PKI infrastructure:
   Router authorization requires certificates
   Managing certificate revocation is complex

3. Limited vendor support:
   Few router vendors implement SEND
   Linux has incomplete SEND support (user-space implementations exist)

4. Interaction with privacy extensions:
   Privacy addresses change frequently
   New CGA must be generated for each privacy address

Practical SEND deployments:
   Limited to specialized environments (military, critical infrastructure)
   Most networks use RA Guard + DHCPv6 Guard instead (simpler, effective)

SEND status check:
   RFC 3971 (SEND) published 2005
   Operational use: rare in production
   Alternative: SeND (Secure NDP) implementations exist as research projects
```

## Conclusion

SEND provides the strongest NDP security through cryptographic address binding and digital signatures, but its complexity, PKI requirements, and computational overhead have limited adoption. For most enterprises, RA Guard at the switch layer combined with DHCPv6 Guard and IPv6 Source Guard provides adequate security without the complexity of SEND. SEND remains the theoretically correct solution for environments requiring proof-of-ownership for IPv6 addresses, particularly in scenarios where switch-level controls cannot be relied upon.
