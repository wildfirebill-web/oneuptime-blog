# How to Interpret ICMP Port Unreachable for UDP Services

Author: [nawazdhandala](https://www.github.com/nawazdhandala)

Tags: ICMP, UDP, Networking, Troubleshooting, Port Unreachable, IPv4

Description: Understand ICMP Port Unreachable (Type 3 Code 3) messages generated when UDP packets reach a host but no service is listening on the target port.

## Introduction

ICMP Port Unreachable (Type 3 Code 3) is generated by a destination host (not a router) when a UDP packet arrives but no application is listening on the target port. Unlike TCP, which uses RST packets to signal a closed port, UDP uses ICMP to report the error. This message is crucial for diagnosing UDP service failures.

## When Port Unreachable is Generated

```
Client sends UDP to 10.20.0.5:514
10.20.0.5 has no syslog daemon running on port 514
10.20.0.5 generates: ICMP Type 3 Code 3 back to client

Key point: Port Unreachable comes from the DESTINATION HOST,
           not from a router. The packet arrived but nothing was listening.
```

## Capturing Port Unreachable Messages

```bash
# Listen for ICMP Port Unreachable
tcpdump -i eth0 -n -v 'icmp[0]=3 and icmp[1]=3'

# Example output:
# 10.20.0.5 > 192.168.1.10: ICMP 10.20.0.5 udp port 514 unreachable, length 36
# -> Nothing listening on UDP 514 at 10.20.0.5
```

## Testing UDP Port Availability

```bash
# Method 1: netcat — attempt UDP connection and check for ICMP error
nc -u -z -v 10.20.0.5 514 2>&1
# "Connection refused" means ICMP Port Unreachable was received
# No response means port is open (UDP doesn't have a handshake) OR filtered

# Method 2: nmap UDP scan
nmap -sU -p 514 10.20.0.5
# "closed" = ICMP Port Unreachable received
# "open" = response received
# "open|filtered" = no response (could be open or firewall blocking)

# Method 3: trigger the error manually with netcat
echo "test" | nc -u 10.20.0.5 12345
# If you get ICMP error back: port is closed
# If no error: either open or firewall is silently dropping
```

## Diagnosing Syslog and DNS UDP Failures

```bash
# Check if syslog is receiving UDP messages (port 514)
# From the syslog server, watch for UDP traffic
tcpdump -i eth0 -n 'udp port 514'

# If you see the UDP packets arriving but get Port Unreachable back:
ss -ulnp | grep ":514"   # Check if syslogd is listening
# If nothing: syslogd not bound to UDP 514, restart it
systemctl restart rsyslog

# For DNS (port 53), check if DNS resolver is running
ss -ulnp | grep ":53"
# If nothing: DNS not running
systemctl start systemd-resolved
```

## Port Unreachable vs Firewall Drop

```bash
# Port Unreachable: you GET an ICMP error back (port is closed, service not running)
# Firewall DROP: no response at all (iptables -j DROP)
# Firewall REJECT with ICMP: you get ICMP back (iptables -j REJECT --reject-with icmp-port-unreachable)

# Distinguish:
# Closed port on reachable host: immediate ICMP error
# Filtered by firewall: timeout (no response)

# Test:
time nc -u -z -w 2 10.20.0.5 514
# Immediate return = closed (ICMP received)
# 2-second timeout = filtered (no response)
```

## Conclusion

ICMP Port Unreachable for UDP is an unambiguous signal that the destination host is reachable but no service is listening on the target port. It's the UDP equivalent of a TCP RST. When you see this message, the fix is to start the expected service on the destination, not to fix routing. When you DON'T see it (timeout instead), investigate whether a firewall is silently dropping the UDP traffic.
